# Альтернативные варианты реализации Яндекс Диска без бэкэнда

## Проблема
Яндекс Диск API не поддерживает CORS для прямых запросов из браузера, поэтому нужен прокси. Текущая реализация через Supabase Edge Function имеет проблемы с обработкой путей.

## Вариант 1: Исправить Edge Function (РЕКОМЕНДУЕТСЯ) ✅

**Преимущества:**
- Полный контроль над навигацией
- Возможность скачивания файлов
- Работает для всех типов файлов

**Что нужно исправить:**
- Правильная обработка путей с кириллицей и пробелами
- Использование относительных путей от корня публичной папки

**Реализация:**
Используем Edge Function как прокси, но с правильной обработкой путей.

## Вариант 2: Использовать прямые публичные ссылки (ОГРАНИЧЕННО)

**Преимущества:**
- Не нужен прокси для просмотра файлов
- Работает напрямую из браузера

**Недостатки:**
- Нужен API для навигации по папкам (все равно нужен прокси)
- Не все файлы имеют прямые публичные ссылки

**Реализация:**
Используем `public_url` из ответа API для прямого открытия файлов, но навигацию делаем через Edge Function.

## Вариант 3: Использовать iframe для встраивания (НЕ РЕКОМЕНДУЕТСЯ)

**Преимущества:**
- Простая реализация

**Недостатки:**
- Ограниченная функциональность
- Нет контроля над интерфейсом
- Проблемы с безопасностью

## Вариант 4: Использовать Supabase RPC функции (АЛЬТЕРНАТИВА)

**Преимущества:**
- Не нужен отдельный Edge Function
- Работает через Supabase напрямую

**Недостатки:**
- Нужно хранить логику в PostgreSQL
- Менее гибко, чем Edge Function

## Рекомендация

**Лучший вариант:** Исправить текущую реализацию Edge Function.

Проблема в том, что путь передается неправильно. Нужно:
1. Правильно нормализовать путь (убрать `disk:` и начальный `/`)
2. Использовать относительный путь от корня публичной папки
3. Правильно кодировать путь с помощью `encodeURIComponent`

## Быстрое решение

Можно использовать прямые публичные ссылки для файлов (если они есть в ответе API), но навигация все равно будет через Edge Function.
